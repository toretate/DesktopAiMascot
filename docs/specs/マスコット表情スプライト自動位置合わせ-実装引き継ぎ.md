## 0. このタスクは何か（30秒サマリ）

マスコット画像から AI 生成した「表情スプライト（最大16種）」を、マスコット本体の顔位置・拡大率に**自動で合わせる**処理を作る。出力は各表情スロットの `offsetX / offsetY / scale / rotation` と顔マスク。

成果物は **`packages/expression-alignment` という framework/DOM 非依存・決定論的なライブラリ**で、UI(Vue)/Server(Express) からは最後に呼び出すだけにする。検証は**テスト（特に正解画像とのIoU）**で行う。

現在はパッケージの**骨格（スキャフォルド）まで完成**。コア本体（レジストレーション）は未実装。

## 1. 確定済みの設計判断（ここが最重要・覆さないこと）

この会話で合意済みの判断。理由付きで列挙する。新規実装者はこれを前提に進めること。

| # | 判断 | 理由 |
|---|---|---|
| D1 | **ローカルonly。AI API（Gemini等）は使わない** | ランニングコスト回避。本処理は設定時の1回限り操作でホットパスではない |
| D2 | **位置合わせは「表情ごと」でなく「シート（衣装×ポーズ）単位で共通」** | 16表情は同一マスコット顔から生成された同一構図。共有変換にしないと表情切替でガタつく |
| D3 | **責務を A（共有変換: scale/rotation/mask）と B（配置: offset）に分離** | 2実行モードを同一関数で扱うため。A はキャッシュ、B は表情ごと |
| D4 | **2実行モード（16枚まとめて / 単体）両対応。共有変換の基準は「通常(neutral)」固定** | 単体実行とまとめ実行の結果を整合させるため。基準以外の単体実行では共有 scale/rotation を上書きしない |
| D5 | **目検出は OpenCV.js の画像レジストレーションが本命**（ORB + `estimateAffinePartial2D` + 補助で `findTransformECC`） | スプライトはマスコット自身の顔から生成された「同一キャラ」。別人検出でなく「同じ顔の2枚の位置合わせ」として解けば、アニメ用ランドマークモデル不在を回避できる |
| D6 | **既存の輝度ベース目検出（feature-island）は主軸から外し、目中心のみのクロスチェック用に縮退** | 閉じ目・ウインク・色付き瞳・肌の濃淡で破綻するため。中心座標は表情で比較的不変なので検証には使える |
| D7 | **face-api.js は不可、transformers.js は弱い** | 対象がアニメ/イラスト顔。face-api.js は実写顔学習。アニメ用ランドマークの ONNX もほぼ無い |
| D8 | **実行は全プラットフォーム on-device（wasm）。サーバ実行は当面不要** | iOS/Android は **Web系スタック（Capacitor / RN+WebView / Flutter+WebView / PWA）で作る前提が確定**。WebView 内で wasm が動くので Electron/Web/モバイルで OpenCV.js コアを100%共有できる |
| D9 | **変換モデルに `rotation` を追加**（`MascotAsset.rotation?: number`、未指定=0で後方互換） | 将来の「横向き・顔の傾き」対応の布石。レジストレーションが回転を自然に出す |
| D10 | **マスクは矩形でなく楕円フェザー（境界ぼかし）** | 顔は丸く、矩形だと角に首・髪・背景が残る |
| D11 | **確信度(confidence)を返し、低い時は既存手動エディタにプリセット値を渡して確認** | 「全自動確定」でなく「自動下書き＋確認」UX |
| D12 | **正解画像 `expr_*_OK.png` とのIoUをしきい値判定する回帰テストを持つ** | パラメータを「測定値」で調整し、リグレッション検出 |
| D13 | **Python anime-face-detector は最終手段（既定では導入しない）** | torch/mmpose は数百MB〜GB級でデスクトップ配布が重い。レジストレーションで不足が確認された横向き等のみ |
| D14 | **コア／アダプタ分離**：コアは `RasterImage`(RGBA) だけ受け取る純粋関数。画像デコードは `ImageLoader` アダプタへ | DOM/実行環境への依存をコアから切り離し、全プラットフォーム共有を可能にする |

## 2. リポジトリ前提（重要）

- **ルートに `package.json` は無い。ワークスペース未構成。** `ui/`（Vue3 + Vite + Vitest、name: `desktop-ai-mascot`）と `server/`（Express + tsx）は**独立**。パッケージマネージャは **npm**。
- そのため `packages/expression-alignment` は**自己完結パッケージ**（独自 package.json / tsconfig / vitest）として作成済み。**ルートワークスペース化はしていない**（重い構造変更になるため）。UI/Server への結線は最後（step9）。
- 既存の同等コードは `ui/src/skills/expression-alignment/` にある（`alignment-calculator.ts` `content-bounds-detector.ts` `face-region-detector.ts` `feature-island-detector.ts` `background-masker.ts` `expression-auto-align.ts`）。**これを移植・改良元にする**。
- ⚠️ 既存コードには **ハードコードされた Windows パス**を `eval("require('c:\\...\\node_modules\\canvas')")` で読む箇所がある（`content-bounds-detector.ts` の `loadImage`、各テスト冒頭）。**移植時に必ず `ImageLoader` 抽象へ置換し、この eval を捨てること**。これが原因で mac 等ではテストが動かなかった。新パッケージは設計で解消済み。

**step3 の設計要点（重要）**：
- 「画像→対応点」（レジストレーション）と「対応点→変換」（数学）を分離した。後者 `solveTransform` は純粋で、対応点を直接与えてテスト済み。
- 前者は **`RegistrationProvider` インターフェース**として注入する（`solveTransform(input, { registration })`）。既定の provider は未実装で `NotImplementedError`（step4 で OpenCV.js 実装を差し込む）。
- `solveTransform` の戻り値は **`SolveTransformResult { transform: SimilarityTransform(px空間), shared: SharedTransform, confidence, method }`**。`transform` はスプライト→ベースの**ピクセル空間**相似変換。**エディタ座標系（offset/scale）への変換は step9（UI結線）で別途**行う（旧コードのプレビュー依存・マジックナンバーをここに持ち込まない）。

**step4 の設計要点（重要・OpenCV のロード）**：
- **`@techstark/opencv-js` は Node/Vitest で await するとハングする**（Emscripten 非同期初期化の既知問題。vitest #3754 / techstark #29、いずれも未解決）。**素の `await cvmod` は CJS/ESM 問わず event loop をブロックして固まる**ことを実機確認済み。
- 解決：**Node/テストでは `opencv-wasm`（同期ロード・初期化待ち不要）を使う**（`adapters/opencv-node.ts`）。ブラウザ/WebView 本番では `@techstark/opencv-js` を `onRuntimeInitialized` で待つ（`adapters/opencv-browser.ts`）。provider は注入された `cv` を受け取るので両対応。
- `opencv-wasm`(4.3.0) に `estimateAffinePartial2D` は無い → **`estimateAffine2D` で RANSAC インライア検出のみ**行い、4自由度の相似変換当てはめは step3 の `similarity.ts` に委譲（責務分離）。`findHomography` / `findTransformECC` も利用可。
- provider は `{ pairs(インライアのみ), inlierRatio }` を返す。`faceRegion` 指定時は base 側に 8UC1 マスクを適用（cv が Rect/Scalar 対応時）。

## 6. 入出力契約（コア API・step3 実装済み）

```ts
solveTransform(
  input: {
    baseImage:   RasterImage;            // マスコット画像（顔クロップ済 or 全身）
    sprite:      RasterImage;            // 通常 or 個別表情（ラベル除去済前提）
    faceRegion?: BoundingBox;            // 既知なら渡す。無ければ provider 内で推定
    sharedTransform?: SharedTransform;   // 単体モードでキャッシュ済みの共有変換A
  },
  deps?: { registration: RegistrationProvider }  // 既定は未実装(NotImplemented)。step4 で注入
): Promise<SolveTransformResult>
```

- `RasterImage = { width, height, data: Uint8ClampedArray /* RGBA, 長さ w*h*4 */ }`
- `RegistrationProvider.register(base, sprite, faceRegion?) → { pairs: PointPair[], inlierRatio }`
  - `PointPair = { src: Point /* スプライト座標 */, dst: Point /* ベース座標 */ }`
- `SolveTransformResult = { transform: SimilarityTransform, shared: SharedTransform, confidence, method }`
  - `SimilarityTransform = { scale, rotation(度), tx, ty }`（スプライト→ベースの**ピクセル空間**。dst = scale·R(rotation)·src + (tx,ty)）
  - `SharedTransform = { scale, rotation, mask?, faceRegion? }`（責務A・シート単位・キャッシュ）
- **エディタ座標系（offsetX/offsetY/scale/rotation, クランプ ±250 / 0.3〜2.0 / ±45）への変換は step9 で別途**。`AlignmentParams` / `ALIGNMENT_CONSTANTS` はそのための型・定数。
- 数学ユーティリティ（`estimateSimilarityTransform` / `estimateTranslation` / `applyTransform` / `residualRms`）も公開済み。

**モード分岐の意味**（実装済み）：
- `sharedTransform` あり（単体・A確立済）→ A の scale/rotation を流用し、平行移動 B（tx,ty）と confidence のみ算出。**shared は渡したものをそのまま返す（D4: A 不上書き）**。
- なし → フル相似変換で A を確立（2点以上必要）。`shared` に scale/rotation を返す。
- 16枚まとめ＝「通常」で A を1回確立し、各表情に sharedTransform=A を渡して B を算出（このループ＝step6）。単体＝A未確立ならその場で確立。
- confidence = clamp(inlierRatio, 0,1) × 残差スコア（残差を dst 点群の対角の10%で正規化）。

## 7. レジストレーションの考え方（D5 の数学）

両目中心（または ORB 対応点群）から相似変換を解く：
- `scale = |両目ベクトル_base| / |両目ベクトル_sprite|`
- `rotation = 2ベクトルの成す角`（顔の傾きに自動対応）
- 平行移動 = 両目中点（or 対応点重心）を一致
- 口を加えた3点以上なら `estimateAffinePartial2D` / 最小二乗で頑健化

**ピクセル空間で純粋に解き、最後にエディタの offset/scale/rotation 値へ変換**する。旧 `alignment-calculator.ts` のプレビュー固定サイズ(420×560)・140px基準・マジックナンバー(`+28/-32/+4`等)依存は持ち込まない（過適合のため）。ORB を主（画風差に強い）、ECC を従（高精度だが見た目依存）。
